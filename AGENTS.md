#### 类型系统

- 不得使用显式 `any`，请尽量使用合理的类型；若确无合理的类型，请使用 `unknown`，此时请使用注释说明理由，格式形如（必须包含`UNKNOWN-TYPE:`）：

  ```typescript

  UNKNOWN-TYPE: 原因总结
  
  原因详细描述
  */
  ```

- 对于泛型，请尽量使用能体现所表示内容的占位符，而不是通用占位符`T`，并同时使用注释说明泛型参数的含义，例如：
  ```typescript
  /**
   * AgentBrain 外部请求契约
   *
   * @template OutputSchema 预期大模型最终返回的强类型结构
   */
  export interface AgentBrainRequest<OutputSchema> {
    context: IContextBuilder; // 使用接口而非具体类，解耦中间层与上下文实现
    workspaceRoot?: string;
    threadId: string;
    skills?: AgentSkillsSignature[]; // 技能签名列表
    tools?: AgentToolSignature[]; // 原子工具签名列表
    outputSchema?: ZodType<OutputSchema>; // Zod 结构校验
    config?: AgentBrainConfig; // 覆盖默认配置
  }
  ```

## 后端

### 代码风格

- Lombok 与依赖注入强制规范：
  - 全面使用 Lombok（如 @Data, @Builder, @Slf4j 等）消除样板代码，禁止手动编写 Getter/Setter 等。
  - 实体类限制：绝对禁止在 JPA 实体类（`@Entity`）上使用 `@Data` 注解。** 为了防止双向关联导致的无限递归（StackOverflow）和意外的懒加载，实体类请仅使用 `@Getter` 和 `@Setter`，并在特殊需要时手动重写 `equals` 和 `hashCode`。
  - 对于实现代码，请不要使用 `@Autowired` 注入依赖，以免降低可测试性，对于测试代码，可以使用 `@Autowired` 简化实现。
  -   所有 Spring Bean使用构造器注入。在类级别使用 `@RequiredArgsConstructor`（若适用），并将所有依赖字段声明为 private final。若需要使用 `@Qualifier`、`@Value` 等参数级别注解，或者构造函数内部包含特殊的初始化/校验逻辑，则可以手动编写构造函数。

#### 类型系统与架构边界

- 请使用 Value Object 表示业务对象的校验状态，例如，校验后的用户名使用单独的 `Username` 类表示，从而避免遗忘校验、多次校验问题；
- 参数校验规范
  - 对于可无状态校验的参数（即，不需要访问数据库、缓存等），应当在 Controller 中校验，相应的 Service 接收校验过的参数；
  - 对于必须有状态校验的参数，不要在 Controller 中引入数据库等，而是在 Service 实现中进行校验。
- Service 应当返回 DTO，而不要直接返回实体。

#### API

- DTO 请创建普通类，并使用 Lombok `@Data` 修饰；
- 请使用 `Jakarta Bean Validation API` 检查用户传入的参数是否符合要求；
- 对于所有业务逻辑，统一使用 `@NonNullApi` 标记，对于可空的参数，再使用 `@Nullable` 标记，尽量不要允许传递空值，可以考虑使用 `Optional`。

## 兼容性

本项目仍然处于初始阶段，不应当存在为了所谓的“兼容性”而引入重复、难以维护代码的决策。

例如，对于下述代码，请直接根据最新文档保留其中的一种，不要为了“兼容”保留两种，增加维护难度。

```typescript
/** 兼容：旧代码使用数组，RFC5 建议使用字典，二者皆支持 */
bug_reports?: BugReport[];
/** RFC5 推荐的字典形式，避免 deepMerge 覆盖问题 */
bug_reports_map?: Record<string, BugReport>;
```
